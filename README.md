## Описание проекта

Современные школы сталкиваются с неэффективностью бумажных систем учёта питания: отсутствие прозрачности, сложность анализа данных и риски безопасности.  
Наше веб-приложение решает эти проблемы, предоставляя **единый цифровой инструмент** для учеников, поваров и администраторов.

Система поддерживает три роли:
- **Ученик**: просмотр меню, оплата (разово/абонемент), указание аллергий, оставление отзывов, отметка получения питания.
- **Повар**: учёт выданных блюд, контроль остатков, создание заявок на закупку.
- **Администратор**: статистика, согласование заявок, формирование отчётов по питанию и затратам.

Проект реализован в соответствии со всеми требованиями регламента испытаний.

## Технологический стек

| Компонент       |   Технология                     |
|------------------|--------------------------------|
| **Frontend**     | React, Vite, CSS               |
| **Backend**      | Django, Django REST Framework   |
| **База данных**  | SQLite (встроенный движок)     |
| **Аутентификация** | JWT (JSON Web Tokens)        |
| **Развёртывание** | Локальное (инструкция ниже)    |

> **Почему SQLite?**  
> Для школьного проекта не требуется масштабируемая СУБД. SQLite — лёгкий, встраиваемый, не требует отдельного сервера, обеспечивает целостность данных и идеально подходит для демонстрации функционала без сложной инфраструктуры.

## Реализованные функции (в соответствии с регламентом)
- Регистрация и авторизация пользователей
- Оплата питания (разовый платёж и абонемент)
- Отметка о получении питания
- Учёт выданных блюд (повар)
- Добавление заявок на закупку (повар)
- Согласование заявок (администратор)
- Формирование отчётов по питанию и затратам
- Обработка исключений (повторная выдача, недостаток порций)
---
## Безопасность
- Все запросы защищены JWT-токенами.
- Персональные данные (аллергии) хранятся только в базе и передаются по защищённым маршрутам.
- Платёжные операции имитируются (без реальных денег), но логика соответствует production-подходу.
